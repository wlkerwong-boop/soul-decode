## 一、现状分析

### 现有资产

| 项目 | 位置 | 说明 |
|------|------|------|
| 灵魂解码 (soul-decode) | `~/Projects/soul-decode` | Next.js 16 + TS + Tailwind + DeepSeek API，aisoulcode.cn |
| 中医通鉴 (tcm-encyclopedia) | `~/Projects/tcm-encyclopedia` | VitePress 静态站，通过 `/tcm/` 子路径部署 |
| 健康评测 /health | soul-decode 内 | 填症状→AI辨证报告，存入 sessionStorage，关闭浏览器即丢失 |
| 八字分析 / (首页) | soul-decode 内 | 输入生日→AI深度报告，也是 sessionStorage |
| 管理员后台 /admin | soul-decode 内 | 管理密码(8888) + 免费码（SOUL-GIFT-xxx），纯前端 |
| 免费码验证 /api/verify-code | soul-decode 内 | API 验证免费码有效性 |
| Supabase 参考 | `~/Projects/my-app/supabase-schema.sql` | stella-aiedu 项目的 Supabase 建表 SQL |

### 关键问题

1. **报告无持久化**：健康评测和八字报告都存 sessionStorage，刷新或关闭即丢失
2. **无用户系统**：只有管理员密码和免费码，用户无法登录、无法看到自己的历史报告
3. **中医通鉴是静态站**：VitePress 生成的纯静态 HTML，不感知用户身份
4. **普光普明的案例**：目前作为公开知识在中医通鉴 dist 中，需要改为管理员私有
5. **技术栈极简**：无数据库、无后端存储、无第三方认证

## 二、架构方案对比

### 方案 A：纯前端 + localStorage/IndexedDB（推荐MVP）

**核心思路**：不引入数据库，利用浏览器本地存储 + 报告ID+密码方式实现"私密空间"

**工作流程**：
1. 用户完成评测/八字分析后，系统生成一个**报告ID**（如 8位 哈希）
2. 用户设置一个**查看密码**（6位数字或自定义密码）
3. 报告 + 元数据（类型、日期、标题）加密后存入 localStorage
4. 用户通过报告ID+密码查看历史报告
5. 管理员预设数据（普光普明）硬编码在代码中，通过管理员标志访问

**优点**：
- 零服务器成本，不改基础设施
- 无需数据库，无需注册流程
- 用户隐私好（数据在本地）
- 一周内可上线

**缺点**：
- 数据仅限同一浏览器（换设备/清缓存就丢）
- 无法跨设备同步
- 无法做"公开贡献"功能（公开到知识库需要服务端）

### 方案 B：Supabase 完整方案（推荐正式版）

**核心思路**：用 Supabase 免费套餐（PostgreSQL + Auth + Storage）建立完整用户系统

**架构**：
```
用户 → 微信小程序/手机验证码登录 → Supabase Auth
                                    → Profiles 表（用户档案）
                                    → Reports 表（健康报告/八字报告）
                                    → Cases 表（管理员预设案例）
                                    → Public contributions 表（用户公开贡献）
```

**数据模型**：
```sql
-- 用户档案
profiles (id, nickname, email/phone, created_at)

-- 报告
reports (id, user_id, type ENUM('health','bazi'), title, content, 
         metadata JSONB, visibility ENUM('private','public'), created_at)

-- 管理员预设案例
admin_cases (id, type ENUM('health_case','bazi_analysis'), author TEXT, 
             title, content, is_published BOOLEAN, created_at)
```

**优点**：
- 真正的用户系统，跨设备、跨浏览器
- 支持"公开贡献"（一键分享到公共知识库）
- 可以用 Supabase Row Level Security 精确控制权限
- 可与中医通鉴联动（通过 API 或 iframe）

**缺点**：
- 需要用户注册/登录（即使简化到微信扫码也要至少一次操作）
- 需要搭建 Supabase 项目（约2-3天）
- 增加了维护成本

### 方案 C：混合方案（推荐过渡期）

**核心思路**：Supabase 做存储后端，但前端用**报告ID + 昵称 + 可选的密码保护**取代完整注册

**工作流程**：
1. 用户在 /health 填完症状提交后，AI 生成报告
2. 系统提示："给这份报告起个名字吧"（如"我的胃胀调理-6月"）
3. 系统生成防伪链接：`aisoulcode.cn/my/report?rid=xxxx`
4. 用户可设置查看密码（可选）
5. 所有报告存 Supabase reports 表，user_id 用 sessionId 或设备指纹关联
6. 用户可以在 /my/profile 看到自己所有报告，通过报告ID+密码/设备指纹访问
7. 如果想跨设备查看 → 设置一个简单的昵称+密码作为"隐形账户"

**优点**：
- 无注册流程，无手机验证
- 数据持久化（即使换浏览器，只要记住报告ID+密码）
- 后端控制权限（管理员可见全部，用户可见自己的）
- 可扩展为完整用户系统
- 中医通鉴的"登录后可查看"可以统一跳转

**缺点**：
- 安全性较弱（报告ID是防伪的但不是强认证）
- 仍有 Supabase 搭建成本

## 三、推荐方案

### MVP阶段（1-2周）→ 方案A（纯前端localStorage + 报告ID/密码）

先让用户能用起来，数据存本地，不丢就行。

### 正式阶段（3-4周）→ 方案C（Supabase + 轻量身份）

等用户量上来，再迁移到云端。

## 七、核心建议

### 先做 MVP（方案A），理由如下：

1. **用户痛点明确** — 报告 sessionStorage 一关就丢，这是最痛点
2. **改动最小** — 不改后端、不引数据库、不增加运维
3. **不影响现有功能** — 健康评测和八字分析流程不变，只是提交后多一步保存
4. **快速验证需求** — 一周内上线，用户用了再说
5. **数据可平滑迁移** — localStorage 的数据结构设计时预留 Supabase 兼容字段，后续迁移无需用户重做

### 具体执行要点：

1. **代码可复用**：report-store.ts 的接口设计成与 Supabase SDK 兼容（saveReport/getReport/getAllReports 签名一致），后续从 localStorage 切到 Supabase 只需改这一个文件
2. **UI 复用现有风格**：/my 页面复用现有的 gradient-bg、input-gold、btn-gold 样式
3. **管理员数据提取**：从 tcm-encyclopedia dist 中提取普光普明案例的 HTML 内容，转成 Markdown 后放入 admin-cases.ts
4. **报告ID生成**：复用现有 `crypto.createHash('md5')` 逻辑（已在 bazi API 中使用），确保同一份输入产生相同的ID
5. **保留扩展点**：在 report 数据结构中预留 `visibility` 字段（private/public），为后续公开贡献做准备

### 不作推荐的事：

- ❌ 不要一开始就上微信登录（增加用户流失）
- ❌ 不要一开始就改中医通鉴的生成逻辑（它是静态站，改起来麻烦）
- ❌ 不要引入支付系统（健康档案是存储功能，不是变现入口）
- ✅ 先在 soul-decode 的 /my 路径下做完，再考虑和中医通鉴的联动
