## 二、根因深度分析

### 2.1 报告字数不足的根因

**文件：** `src/app/api/master-report/route.ts`

```
max_tokens: 6000,
temperature: 0.7,
```

**问题链：**
1. `max_tokens: 6000` — DeepSeek V4 Pro对中文的单Token约覆盖1.5-2字，6000 tokens ≈ 3000-4000字。即使用到上限也只有4000字，离5000-10000相差甚远。
2. **但更关键的是：** 即使token够，输出也只用了2850字，说明prompt本身没有鼓励模型写到上限。模型根据指令长度和质量自动推断输出长度。
3. **prompt总输入长度** — 用户数据只有几十行，指令只有数段，模型接收到的是一个"短prompt→短输出"的暗示。
4. 对比Kimi：Kimi的prompt通常包含非常详细的section-level字数要求，如"每个分析维度不少于300字"，从而每段都有最低产出保证。

### 2.2 人类图数据缺失的根因

**文件分析：**

| 路由 | 使用的HD引擎 | 外部依赖 | 在Vercel上 |
|------|------------|---------|-----------|
| `fusion/route.ts` | `hd-engine-v5.cjs` | **零外部依赖**，全内联星历表 | ✅ 可工作 |
| `master-report/route.ts` | `hd-engine-v6.cjs` | `@fusionstrings/swisseph-wasm` + `@it-healer/human-design-calculator` | ❌ 静默失败 |

**失败原因（3个致命点）：**

1. **`installCommand: "npm install --ignore-scripts"`**（vercel.json第2行）
   - swisseph-wasm 需要在安装时运行脚本构建WASM二进制文件
   - `--ignore-scripts` 跳过了这个关键步骤
   - 导致WASM模块加载时抛异常：`TypeError: WebAssembly.instantiate is not a function` 或 二进制文件404

2. **异常被静默吞掉**
   ```typescript
   function calcHD(...) {
     try {
       const mod = require('../../../lib/hd-engine-v6.cjs');
       return mod.calculateBodygraph(...);
     } catch { return null; }  // ← 静默返回null，前端不知道出错了
   }
   ```
   在 `master-report/route.ts` 第23行，`catch { return null; }` 完全屏蔽了错误信息。
   而第121-130行在return时：
   ```
   ${hdResult ? `【二、人类图】...` : ''}
   ```
   hdResult为null时，整个人类图章节被完全跳过。

3. **时区/坐标参数问题**
   - v6引擎调用：`calculateBodygraph(ds, ts, tz, 0, 0)` — 经纬度传入了(0,0)
   - 这在某些swisseph版本中会导致计算异常
   - v5引擎有类似的`lat, lon`参数但内部不使用它

### 2.3 内容泛泛的根因

**当前system prompt（master-report/route.ts第160行）：**
```
system: '你是一位修行数十年的命理导师，精通八字、人类图、西方占星、紫微斗数、五运六气五大体系。
        你的报告温暖、精准、有深度，像长辈谈心一样自然。从不使用AI套话。'
```

**问题：** 过于简短，缺少以下关键元素：
- ❌ 无具体输出格式规范（标题层级、分段、强调方式）
- ❌ 无字数约束（每部分至少多少字）
- ❌ 无推理步骤要求（"先分析数据，再交叉验证，最后输出结论"）
- ❌ 无示例标杆（Kimi做得好是因为有大量文风示例）
- ❌ 无反例说明（"避免以下AI套话：'值得注意的是'、'让我们来看看'、'在当今社会'"）
- ❌ 无数据驱动的深度要求（"必须引用具体数据点，不能只说抽象概念"）

**对比当前fusion/route.ts** — 虽然也是三系统，但prompt结构更详细，包含了：
- 禁止使用的套话清单 ✓
- 各维度具体内容要求 ✓
- 风格标杆 ✓

但fusion route使用的是旧model：`deepseek-chat`（不是V4 Pro），且`max_tokens: 4000`（更少）。

## 三、优化方案

### 3.1 Prompt 结构重构（最高优先级）

#### 3.1.1 System Prompt 重写

```typescript
export const MASTER_REPORT_SYSTEM_PROMPT = `你是修炼数十年的命理导师，精通八字命理、人类图(Human Design)、西方占星、紫微斗数、五运六气、流年运势、人生规划七大体系。

【核心原则】
- 你写的是「人生传记」而非「算法报告」——每个数据点都要转化成具体的人生场景
- 你必须找到七系统之间的内在交叉印证，不能分段罗列
- 你的语言要像长辈跟孩子谈心——温暖、直接、有力，绝不回避尖锐话题

【输出规范】
- 总字数：5000-8000字
- 使用 Markdown 排版，有明确的层级结构
- 每个分析维度不能少于400字
- 每个维度结尾必须有一句「可以立即执行的具体建议」
- 最后必须有「一句话点睛」

【写作范例】
✅ 好的写法：
「你的八字日主庚金，生于丑月，金寒水冷。这让你天生有一种'疏离感'——不是冷漠，而是你总是在热闹中抽离出来观察。人类图中你是Projector（投射者），'等待被邀请'是你的策略。这两者互相印证：你不需要主动融入每个圈子，你的天赋是'被请入'一个需要你洞察力的位置。」

❌ 差的写法：
「日主庚金代表刚毅果断，人类图类型为投射者。建议在工作中发挥优势。」（这是AI套话，没有任何个人针对性）

【铁律】
1. 禁止使用以下套话开头：「让我们来看看」「首先」「其次」「在当今这个时代」「值得一提的是」
2. 禁止使用以下空洞总结：「总的来说」「综上所述」
3. 每个数据必须对应具体人生场景，不能说抽象概念
4. 必须对数据不足的部分诚实标注「基于现有信息」或「以下为推测」`;
```

### 3.2 max_tokens 和 API 参数优化

| 参数 | 当前值 | 优化值 | 说明 |
|------|-------|-------|------|
| `max_tokens` | 6000 | **12000-16000** | 5000-8000字×2字/token≈10000-16000 tokens |
| `temperature` | 0.7 | **0.8-0.85** | 提高创造性，使语言更鲜活、有画面感 |
| `top_p` | 未设置 | **0.9** | 增加词汇多样性 |
| `frequency_penalty` | 未设置 | **0.3** | 减少重复用词 |
| `presence_penalty` | 未设置 | **0.2** | 增加内容覆盖面 |
| `stream` | false | **true** | 流式输出可提升用户体验，同时也鼓励模型持续产出 |

> **注意：** max_tokens每增加10000，费用增加。10000 tokens × $0.014（DeepSeek V4 Pro输出价格）= $0.14/次。若每天100份报告，约$14/天。如预算有限，可降为10000 tokens，7000字左右，与Kimi相当。

### 3.3 人类图数据修复方案

**推荐方案（立即生效，零风险）：**

将 `master-report/route.ts` 的 `calcHD` 函数从v6引擎切换到v5引擎：

```typescript
// 修改前（第17-24行）：
function calcHD(y, m, d, h, tz) {
  try {
    const mod = require('../../../lib/hd-engine-v6.cjs');  // ← 有WASM依赖
    ...
  } catch { return null; }
}

// 修改后：
function calcHD(y, m, d, h, tz) {
  try {
    // 使用v5引擎（零外部依赖，Vercel兼容）
    const mod = require('../../../lib/hd-engine-v5.cjs');
    const ds = `${String(y).padStart(4,'0')}-${String(m).padStart(2,'0')}-${String(d).padStart(2,'0')}`;
    const ts = `${String(h).padStart(2,'0')}:00`;
    return mod.calculateBodygraph(ds, ts, tz || 'Asia/Shanghai', 39.9, 116.4);
  } catch (e) {
    console.error('[HD] 人类图计算失败:', e.message);
    return null;
  }
}
```

**长期方案（精度提升）：**
1. 将v5引擎的星历表从JSON文件改为内联（消除文件读取依赖）
2. 对比v5与鼎新图的闸门匹配度（目前16个目标中匹配13个）
3. 修正v5中Design日计算（从固定88天改为太阳弧精确计算）
4. 添加时区转换修正（参考fusion/route.ts中的offsetMap逻辑）

### 3.4 多系统交叉推理增强

**目前的问题：** 报告是把七系统数据一股脑塞给AI，依赖AI自己找交叉点。

**优化方案 — 增加「推理步骤」到prompt中：**

```typescript
【推理步骤要求】
在正式撰写报告之前，请先在内部完成以下分析（不需要输出到报告）：
1. 逐一分析每个系统的核心信号（每个系统只提取3-5个最关键的信号）
2. 标记所有信号之间的「印证关系」和「矛盾关系」
3. 找出重复出现2次以上的信号（这些是核心主题）
4. 对矛盾信号给出解释（为什么系统A这样说，系统B那样说？）
5. 基于以上分析，确定报告的核心叙事线

只有在完成上述推理后，才开始撰写正式报告内容。
```
